<objective>
Document the `RELEASE_TOKEN` PAT prerequisite in `docs/RELEASING.md` and gate phase completion on a human checkpoint that confirms the secret exists in repository settings. Without the secret, the entire Phase 21 scaffold is silently a no-op — a published release would never trigger `publish-packages.yml`. This is the only step that genuinely requires human action (creating a PAT in a GitHub UI is not automatable from CI), so it is correctly modeled as a checkpoint, not a script.

(2) Enforce the precondition by pausing for human confirmation that the secret exists before the phase is considered complete.

Output: New file `docs/RELEASING.md` (alongside the existing `docs/JSR_SETUP.md` operational doc, same shape) AND a confirmed-exists `RELEASE_TOKEN` repo secret.

Create a new file `docs/RELEASING.md`. The file does not exist yet — use the Write tool. Match the shape and audience of `docs/JSR_SETUP.md` (a maintainer-facing operational doc). Suggested structure (write your own prose; do not just paste this skeleton):

1. **Title and one-line purpose**: `# Releasing nsyte` and a single sentence stating this doc covers the prerequisites and recovery for the package-manager publish workflow.

2. **`## RELEASE_TOKEN prerequisite`** — the most important section.
   - State plainly: a Personal Access Token named `RELEASE_TOKEN` MUST be configured as a repository secret.
   - Explain WHY (one short paragraph): GitHub suppresses workflow events triggered by `GITHUB_TOKEN`-initiated actions. A release created with `GITHUB_TOKEN` does not emit `release: published` to other workflows. A user-classed PAT bypasses this suppression. Cite the consumer: `.github/workflows/release.yml` line 446 uses this token; `.github/workflows/publish-packages.yml` triggers on the resulting event.
   - State the SYMPTOM of forgetting it: a tag push or workflow_dispatch creates a release normally, but `publish-packages.yml` never appears in the Actions tab. No package managers update.

3. **`## Creating the PAT`** — exact steps.
   - Option A (classic): Navigate to https://github.com/settings/tokens → Generate new token (classic). Required scope: `repo`. Set an explicit expiry (90 days recommended, max 1 year — note that an indefinite token is a higher security risk).
   - Option B (fine-grained, preferred): Navigate to https://github.com/settings/personal-access-tokens → Generate new token. Repository access: only `sandwichfarm/nsyte`. Repository permissions: `Contents: Read and write`. (No org or account permissions needed.)
   - Either option produces a token string starting with `ghp_` (classic) or `github_pat_` (fine-grained). Copy it once — GitHub never shows it again.

4. **`## Adding the secret to the repository`**.
   - Navigate to https://github.com/sandwichfarm/nsyte/settings/secrets/actions.
   - Click "New repository secret".
   - Name: `RELEASE_TOKEN` (exactly — case-sensitive). Value: the PAT string from above.
   - Click Add secret. The secret name will appear in the list; the value is never visible again.

5. **`## Verifying the setup`**.
   - Manual: in repo Settings → Secrets and variables → Actions, confirm `RELEASE_TOKEN` is listed.
   - Functional: trigger `release.yml` via `workflow_dispatch` against a test version (use the `draft: true` option to avoid publishing a real release), then convert the draft to a published release, then watch the Actions tab — `publish-packages.yml` should appear within ~30 seconds and run the setup job.

6. **`## Rotation`**.
   - PATs eventually expire (or get revoked). When `RELEASE_TOKEN` expires, the next release silently fails to trigger `publish-packages.yml`.
   - Set a calendar reminder for the day before the expiry date.
   - To rotate: create a new PAT (same scope), update the `RELEASE_TOKEN` secret value (same name), test with a draft release.

7. **`## Troubleshooting`**.
   - Symptom: `publish-packages.yml` never appears after a release.
     Causes (in order of likelihood): (a) `release.yml` step still references `GITHUB_TOKEN` instead of `RELEASE_TOKEN` — re-check line 446; (b) `RELEASE_TOKEN` secret not set; (c) `RELEASE_TOKEN` expired or revoked; (d) PAT lacks `repo` / `Contents: write` scope.
   - Symptom: `publish-packages.yml` setup job fails on the asset-readiness poll with "Release assets not available after 300s".
     Causes: `release.yml` build jobs failed (check `release.yml` run for that tag); release was published before assets finished uploading (rare with softprops/action-gh-release); asset names changed in `release.yml` and the poll's expected list is stale (update poll's expected names in sync with `release.yml` output).

8. **`## Related files`**.
   - `.github/workflows/release.yml` — the release-creation workflow; uses `RELEASE_TOKEN` on the `softprops/action-gh-release` step.
   - `.github/workflows/publish-packages.yml` — the publish workflow; triggered by the PAT-created release.

Constraint: Do not include the actual token value, screenshots of the GitHub UI with tokens visible, or example tokens that look real. Use placeholders like `<your-pat-here>` if a token must be shown in syntax.
