Replace the Phase 21 stub `publish-aur` job in `.github/workflows/publish-packages.yml` with the real publish pipeline. The new job: loads the AUR SSH key into an ssh-agent, pins the AUR host key, clones the AUR git remote, copies the PKGBUILD template from the repo, sed-substitutes the named placeholders (version + per-arch SHA256), regenerates `.SRCINFO` inside an Arch Linux container as a non-root user, then commits and pushes both files atomically to the AUR remote in a single step.

Purpose: AUR `nsyte-bin` ships and auto-updates on every GitHub release with no manual maintainer intervention. Per-arch checksums are mapped by name (not array index), so future arch additions cannot silently mis-assign hashes. The `.SRCINFO` is always regenerated from the patched `PKGBUILD` via `makepkg --printsrcinfo` so it can never drift from the canonical metadata. The push is atomic — AUR rejects desynced PKGBUILD/.SRCINFO pairs, so both files land in one commit.

```yaml
  publish-aur:
    name: Publish to AUR
    needs: [setup]
    if: needs.setup.result == 'success'
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - name: Checkout nsyte repo (for PKGBUILD template)
        uses: actions/checkout@v4

      - name: Set up SSH agent with AUR key
        uses: webfactory/ssh-agent@v0.10.0
        with:
          ssh-private-key: ${{ secrets.AUR_SSH_PRIVATE_KEY }}

      - name: Add AUR host to known_hosts
        run: |
          mkdir -p ~/.ssh
          chmod 700 ~/.ssh
          ssh-keyscan aur.archlinux.org >> ~/.ssh/known_hosts

      - name: Clone AUR remote
        run: |
          git clone ssh://aur@aur.archlinux.org/nsyte-bin.git /tmp/aur-nsyte-bin
          # Note: "You appear to have cloned an empty repository" on first run is
          # the documented bootstrap path. The first push creates the AUR package.

      - name: Patch PKGBUILD from template
        env:
          VERSION: ${{ needs.setup.outputs.version }}
          SHA256_X86_64: ${{ needs.setup.outputs.sha256_linux_x64 }}
        run: |
          cp packages/aur/PKGBUILD /tmp/aur-nsyte-bin/PKGBUILD
          sed -i "s/PLACEHOLDER_VERSION/${VERSION}/g"             /tmp/aur-nsyte-bin/PKGBUILD
          sed -i "s/PLACEHOLDER_SHA256_X86_64/${SHA256_X86_64}/g" /tmp/aur-nsyte-bin/PKGBUILD
          # Sanity: fail fast if any placeholder survived substitution
          if grep -q "PLACEHOLDER_" /tmp/aur-nsyte-bin/PKGBUILD; then
            echo "::error::PKGBUILD still contains PLACEHOLDER_ markers after substitution."
            grep "PLACEHOLDER_" /tmp/aur-nsyte-bin/PKGBUILD
            exit 1
          fi

      - name: Regenerate .SRCINFO in Arch Linux container
        run: |
          docker run --rm \
            -v /tmp/aur-nsyte-bin:/pkg \
            archlinux:base-devel \
            bash -c '
              set -euo pipefail
              pacman -Sy --noconfirm 2>/dev/null || true
              useradd -m builder
              chown -R builder /pkg
              cd /pkg
              sudo -u builder makepkg --printsrcinfo > .SRCINFO
            '
          # Sanity: ensure .SRCINFO actually landed on the host filesystem
          test -s /tmp/aur-nsyte-bin/.SRCINFO || {
            echo "::error::.SRCINFO was not generated or is empty."
            exit 1
          }

      - name: Commit and push to AUR
        env:
          VERSION: ${{ needs.setup.outputs.version }}
        run: |
          git -C /tmp/aur-nsyte-bin config user.name "nsyte-bot"
          git -C /tmp/aur-nsyte-bin config user.email "sandwich.farm@protonmail.com"
          git -C /tmp/aur-nsyte-bin add PKGBUILD .SRCINFO
          git -C /tmp/aur-nsyte-bin commit -m "chore: update to v${VERSION}"
          git -C /tmp/aur-nsyte-bin push origin master
```

**Critical correctness rules — these are not stylistic; deviating breaks the job:**

1. **Indentation is two spaces per level.** The `publish-aur:` job key sits at the same indent as `setup:`, `publish-homebrew:`, etc. (i.e., two spaces from line start, since they live under the top-level `jobs:` key).

2. **Use `env:` blocks, not inline `${{ }}` in shell, for VERSION and SHA256_X86_64.** Inline `${{ secrets.AUR_SSH_PRIVATE_KEY }}` in a `run:` block would print the secret's substituted form into the shell command and break ssh-agent setup. Inline VERSION/SHA256 is also a quoting hazard. The `env:` pattern (used in setup job's `poll` step at line 41-42) is the established repo convention.

3. **Single quotes around the bash heredoc body in the docker step (`bash -c '...'`).** Double quotes would let the host shell interpolate `$pkg`, `$builder`, etc. before the container sees them. Single quotes pass the script through untouched. Inside the single-quoted body, do NOT use any `${{ }}` expressions — they would not interpolate inside single quotes anyway, but the failure mode is silent.

4. **`set -euo pipefail` inside the docker bash script.** Without it, a failure of `useradd` or `chown` continues silently and the makepkg failure surfaces with no useful context.

5. **Use `archlinux:base-devel`, NOT `archlinux:latest`.** The `:latest` tag is the minimal base image without `base-devel`; `makepkg` lives in `base-devel` (RESEARCH.md Anti-Patterns, Pitfall: pacman database). The PATTERNS doc and CONTEXT.md briefly mention `:latest` — they are wrong on this point; RESEARCH.md and Open Question #3 corrected to `:base-devel`. Use `:base-devel`.

6. **`cd /pkg` BEFORE the `>` redirect.** The redirect target is relative to the working directory at the moment `makepkg --printsrcinfo > .SRCINFO` runs. Without `cd /pkg`, `.SRCINFO` lands in `/home/builder` (or wherever sudo -u resets cwd to) and is invisible to the host filesystem (Pitfall 5).

7. **Stage `PKGBUILD` and `.SRCINFO` in the same `git add`.** AUR rejects pushes where the two files are out of sync. The single `git add PKGBUILD .SRCINFO` followed by one commit guarantees atomicity.

8. **Push to `origin master`, not `origin main`.** AUR uses `master` as its default branch — verified by AUR submission guidelines.

9. **Do NOT add `actions/checkout` to fetch the AUR repo.** The AUR remote uses SSH, not HTTPS, and needs the ssh-agent set up first. Use plain `git clone` after the ssh-agent step.

10. **Do NOT introduce any new third-party action besides `webfactory/ssh-agent@v0.10.0`.** RESEARCH.md "Don't Hand-Roll" justifies why we keep the rest as shell. No `KSXGitHub/github-actions-deploy-aur`, no `aurpublish` action.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner -> GitHub Releases | Read-only via `actions/checkout` for the nsyte template; no writes to nsyte repo from this job. |
| GitHub Actions secrets store -> ssh-agent | `AUR_SSH_PRIVATE_KEY` is loaded into the runner's ssh-agent via `webfactory/ssh-agent@v0.10.0` and never written to disk. |
| GitHub Actions runner -> AUR git remote | Authenticated SSH push using `AUR_SSH_PRIVATE_KEY`. The AUR remote is the trust sink — once we push, the AUR package page is updated automatically. |
| Setup job outputs -> shell sed | `version` and `sha256_linux_x64` cross from a controlled CI job into a shell `sed` command. Values are validated upstream (semver pattern; 64-char hex). |
| Docker container -> runner filesystem | The container runs as root inside its namespace but only writes to the bind-mounted `/tmp/aur-nsyte-bin`. The host's ownership is restored after the run by writing as `builder` (UID created inside container) — host runner only reads the resulting file. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-01 | Information Disclosure | `AUR_SSH_PRIVATE_KEY` value leaking via step logs | mitigate | Use `webfactory/ssh-agent@v0.10.0` (loads key only into ssh-agent socket; never writes to disk; auto-masks output). Never `echo` the secret. The action's release notes explicitly call out log scrubbing. |
| T-22-02 | Tampering | Shell injection via `version` or `sha256_linux_x64` setup outputs | mitigate | Both values are bound to step `env:` (`VERSION`, `SHA256_X86_64`), then expanded as `"$VERSION"` / `"$SHA256_X86_64"` in `sed` commands. Setup job upstream guarantees `version` is `${TAG#v}` from a tag pushed to the repo (already privileged-write gated) and `sha256_linux_x64` is the deterministic hex output of `sha256sum` against a downloaded asset. No user-supplied unconstrained string reaches `sed`. |
| T-22-03 | Spoofing | MITM on `ssh-keyscan aur.archlinux.org` resolving to a malicious host | accept | The push payload (PKGBUILD + .SRCINFO) is public information — there is no secret to steal. A successful MITM would only be able to misroute the push, which would cause a visible CI failure (mismatched key) rather than silent compromise. RESEARCH.md security domain explicitly accepts this tradeoff for AUR (Pattern 2 + Security Domain table). |
| T-22-04 | Tampering | Malicious `archlinux:base-devel` Docker Hub image substitution | accept | We pull the image at runtime from Docker Hub, the official Arch Linux organization. Pinning a digest would harden this further, but the impact of a compromised base image in this job is limited: the only writes from inside the container go to `/pkg` (the AUR clone) and the only command run there is `makepkg --printsrcinfo`, whose output is committed verbatim to a public AUR repo. RESEARCH.md security domain explicitly accepts this risk. |
| T-22-05 | Elevation of Privilege | `makepkg` running as root inside the container | mitigate | The docker step creates a non-root `builder` user via `useradd -m builder`, chowns the volume to `builder`, and runs `makepkg --printsrcinfo` via `sudo -u builder`. This is a hard requirement — `makepkg` refuses to run as root and will exit with an error. Mitigation is correctness, not just hardening. |
| T-22-06 | Repudiation | AUR push attributed to wrong author | accept | `git config user.email "sandwich.farm@protonmail.com"` and `user.name "nsyte-bot"` are set on the AUR clone. AUR commit history shows nsyte-bot as author. This is acceptable; the commit signature is the SSH key fingerprint, which is auditable in AUR account settings.
