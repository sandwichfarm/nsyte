**Workflow name and triggers** (per INFRA-02):
- `name: publish packages`
- `on:`
  - `release: types: [published]`
  - `workflow_dispatch:` with input `tag` (string, required, description "Release tag to publish (e.g. v1.6.0)")

**Setup job** (`setup`, runs-on `ubuntu-latest`, per INFRA-04):

Declare job-level `outputs:` mapping these five keys to their step outputs:
- `version`            -> `${{ steps.extract.outputs.version }}`
- `sha256_linux_x64`   -> `${{ steps.checksums.outputs.sha256_linux_x64 }}`
- `sha256_macos_arm64` -> `${{ steps.checksums.outputs.sha256_macos_arm64 }}`
- `sha256_macos_x64`   -> `${{ steps.checksums.outputs.sha256_macos_x64 }}`
- `sha256_windows`     -> `${{ steps.checksums.outputs.sha256_windows }}`

Steps in order:

1. **Step `extract` (id: extract)**: bash run step.
   - Assign tag from coalesced expression to a bash variable (NEVER inline ${{ }} into shell commands — security: shell injection via workflow_dispatch input):
     ```bash
     TAG="${{ github.event.release.tag_name || github.event.inputs.tag }}"
     VERSION="${TAG#v}"
     echo "version=${VERSION}" >> "$GITHUB_OUTPUT"
     echo "tag=${TAG}" >> "$GITHUB_OUTPUT"
     ```
   - Output `tag` is needed by the poll step. Output `version` is the public API consumed by manager jobs.

2. **Step `poll` (id: poll)**: bash run step with `env: GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}`.
   - Reads `${{ steps.extract.outputs.tag }}` and `${{ steps.extract.outputs.version }}` into bash vars (quote them).
   - Loops with INTERVAL=15 and TIMEOUT=300.
   - Each iteration: `NAMES=$(gh release view "$TAG" --json assets --jq '.assets[].name' 2>/dev/null || echo "")`
   - Expected list (string): `EXPECTED="nsyte-linux-${VERSION} nsyte-macos-arm64-${VERSION} nsyte-macos-x64-${VERSION} nsyte-windows-${VERSION}.exe"`
   - Sets `ALL_PRESENT=true`, then iterates `for expected in $EXPECTED` and uses `echo "$NAMES" | grep -qF "$expected"` to detect presence; on miss, sets ALL_PRESENT=false and breaks the inner loop.
   - On all-present, breaks the outer loop. Otherwise sleeps `$INTERVAL` and accumulates `ELAPSED`.
   - On timeout exit: `echo "::error::Release assets not available after ${TIMEOUT}s. Aborting."` then `exit 1`.
   - On success: `echo "All expected assets present after ${ELAPSED}s."`

3. **Step `checksums` (id: checksums)**: bash run step.
   - Reads version (no `v` prefix) into bash var. Constructs `BASE="https://github.com/${{ github.repository }}/releases/download/v${VERSION}"`.
   - `curl -fsSL` each of the four standard binaries to local files:
     ```bash
     curl -fsSL -o nsyte-linux       "${BASE}/nsyte-linux-${VERSION}"
     curl -fsSL -o nsyte-macos-arm64 "${BASE}/nsyte-macos-arm64-${VERSION}"
     curl -fsSL -o nsyte-macos-x64   "${BASE}/nsyte-macos-x64-${VERSION}"
     curl -fsSL -o nsyte-windows.exe "${BASE}/nsyte-windows-${VERSION}.exe"
     ```
   - Computes hex SHA256 for each, writes to GITHUB_OUTPUT in a single grouped redirect:
     ```bash
     {
       echo "sha256_linux_x64=$(sha256sum nsyte-linux       | awk '{print $1}')"
       echo "sha256_macos_arm64=$(sha256sum nsyte-macos-arm64 | awk '{print $1}')"
       echo "sha256_macos_x64=$(sha256sum nsyte-macos-x64   | awk '{print $1}')"
       echo "sha256_windows=$(sha256sum nsyte-windows.exe   | awk '{print $1}')"
     } >> "$GITHUB_OUTPUT"
     ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner -> GitHub Releases API | Read-only asset enumeration via `gh` CLI, authenticated by built-in `GITHUB_TOKEN`. No write actions in this scaffold. |
| `workflow_dispatch` input -> shell command | User-supplied `tag` input is interpolated into a bash variable, then quoted. Direct interpolation into a command line would be a shell-injection vector. |
| Secrets store -> step env | Only `GITHUB_TOKEN` is consumed in this phase, on a read-only path. Manager-specific secrets (`AUR_SSH_KEY`, etc.) are introduced in phases 22-26. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-04 | Tampering | Shell injection via `workflow_dispatch.inputs.tag` | mitigate | The `tag` input is assigned to a bash variable (`TAG="${{ ... }}"`) then used only as `"$TAG"` in subsequent commands. Never interpolated inline into a `gh` or `curl` command. Per V5 control. |
| T-21-05 | Information Disclosure | Asset URL or version leaking sensitive context | accept | All asset URLs and the version string are public — they appear in published GitHub Releases. No PII or secret material is emitted to logs by this scaffold. |
| T-21-06 | Denial of Service | Asset poll never converges, holding a runner for 5 minutes | accept | Poll has hard 300s timeout and exits 1 — bounded resource use. Failure mode is loud (workflow turns red), not silent. |
| T-21-07 | Spoofing | Asset name collision (a malicious release with attacker-named asset) | accept | The release is created by the same repo's `release.yml` (gated by tag-push permissions). The `nsyte-linux-{VERSION}` naming is owned by us. Out-of-band asset injection requires repo write access — already a higher threat than this workflow. |
| T-21-08 | Information Disclosure | `GH_TOKEN` value leaking via gh CLI verbose mode | mitigate | gh is invoked without `--verbose` or `-v`. GitHub Actions auto-masks `${{ secrets.* }}` values in step output. |
| T-21-09 | Elevation of Privilege | Manager job stubs running with broader permissions than needed | accept | Stubs run on a default-permission runner with no checkout, no token, no secret access. They `echo` and `exit 0`. Phase 22+ will scope tokens per-manager when introducing real publish logic. |
</threat_model>
